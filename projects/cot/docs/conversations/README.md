# 对话记录约定 (Conversation Logging Convention)

> **创建日期**: 2026-04-08  
> **创建人**: why (用户) + Kimi Claw (AI Assistant)  
> **适用范围**: Project Cot (缅甸光储柴微网 EMS 项目)

---

## 1. 为什么需要这个约定

AI 助手在会话压缩后会丢失详细对话内容，只能保留摘要。为了：
- 不遗忘项目关键决策
- 追溯技术方案演进过程
- 让未来的自己能理解过去的上下文

用户 why 要求建立此机制。

## 2. 记录规则

### 2.1 记录范围
**所有涉及 Cot 项目的对话**必须记录：
- PRD 文档讨论
- 架构设计讨论
- 技术方案决策
- 需求变更确认
- 问题排查过程

### 2.2 记录位置
```
projects/cot/docs/conversations/
├── 2026-04-08.md      # 按日期命名
├── 2026-04-09.md      # 新的一天 = 新文件
└── README.md          # 本文件（约定说明）
```

### 2.3 记录格式
每个日期文件包含：
1. **会话概览** - 日期、参与人、主题
2. **对话详情** - 按时间顺序记录关键对话
3. **关键决策** - 技术方案、参数选择
4. **生成的文件** - 当天创建/修改的文档
5. **待确认事项** - 遗留问题、TODO

### 2.4 何时记录
- **实时**: 对话进行中，关键决策立即记录
- **结束**: 会话结束前，整理完整记录
- **压缩后**: 如果会话被压缩，基于摘要补充关键信息

## 3. 如何查阅

### 3.1 启动时必读
每次关于 Cot 项目的对话开始前，AI 必须：
1. 读取本文件 (README.md) 确认约定
2. 读取当天的日期文件 (如 2026-04-08.md)
3. 读取最近 3 天的历史记录

### 3.2 关键记忆点（防止犯错）

**每次启动Cot项目对话前，必须做以下检查**：

1. **拓扑纠正检查** ⚠️
   - 读取 `02-Domain-Model.md` 第2.1节拓扑图
   - 确认理解：MPPT直流输出直接到直流母线，不经过PCS
   - 确认理解：PCS只负责AC/DC变换，不是光伏到电池的必经之路

2. **然后继续正常流程**：
   - 读取当天的日期文件
   - 读取最近3天的历史记录

**为什么这很重要**：
我曾经多次误以为"光伏通过PCS充电到电池"，这是严重的错误。正确的理解是：光伏直接输出直流电到直流母线，与电池并联，PCS只负责交流侧和直流侧的AC/DC变换。

## 5. 项目文档结构定义 (ai-dev/)

Cot 项目的核心文档位于 `projects/cot/docs/ai-dev/`，采用 00~08 编号体系。每次启动必须确认此结构存在。

### 5.1 文档清单

| 编号 | 文件名 | 用途 | 为什么需要 |
|------|--------|------|-----------|
| 00 | `00-Context.md` | 项目上下文速查 | 快速了解项目背景、术语、约束，无需翻阅完整PRD |
| 01 | `01-PRD.md` | 产品需求文档 | 定义功能需求(FR-XXX)、用户场景、验收标准 |
| 02 | `02-Domain-Model.md` | 领域模型 | 数据实体、设备模型、通信协议映射（南向/北向） |
| 03 | `03-Functional-Spec.md` | 功能规格说明书 | 详细功能设计、算法逻辑、控制策略实现细节 |
| 04 | `04-State-Machine.md` | 状态机定义 | EMS核心状态流转（并网/离网/柴发等模式切换） |
| 05 | `05-Technical-Design.md` | 技术设计文档 | 架构设计、模块划分、接口定义、技术选型 |
| 06 | `06-API-Contract.yml` | 接口契约 | 前后端/设备间接口规范、数据格式、协议定义 |
| 07 | `07-Test-Spec.md` | 测试规格 | 测试用例、验收条件、边界条件、故障注入场景 |
| 08 | `08-Task-Breakdown.json` | 模块化任务清单 | 开发任务拆分、依赖关系、工时估算、责任人 |

### 5.2 为什么是这个结构

**编号而非命名**：强制优先级顺序，00→08代表从宏观到微观、从需求到实现的逻辑流。

**分层设计**：
- 00~01：What（是什么）
- 02~03：How（怎么做）
- 04~05：Implementation（如何实现）
- 06~08：Delivery（交付验证）

**记忆锚点**：
- 00=Context（上下文）
- 01=PRD（需求）
- 02=Domain（领域）
- 03=Func（功能）
- 04=State（状态机）
- 05=Tech（技术）
- 06=API（接口）
- 07=Test（测试）
- 08=Task（任务）

### 5.3 启动自检
每次启动时确认：
- [ ] 00-Context.md 存在且非空
- [ ] 01-PRD.md 版本号 ≥ 1.5
- [ ] 02-Domain-Model.md 包含南向设备协议
- [ ] 03~08 文档骨架存在（即使内容待填充）
- [ ] PENDING-ITEMS.md 存在（待确认事项汇总）

如缺失，询问用户是否生成骨架文件。

### 5.4 ⚠️ 关键自检：拓扑结构理解（必读）

**这是极其重要的检查项 - 我多次犯过的错误**

每次处理Cot项目时，**必须**先读取并确认正确的功率拓扑：

```
┌─────────────────────────────────────────┐
│  ⚠️  禁止的错误理解                      │
│  光伏 → PCS → 电池（交流路径）           │
│                                          │
│  ✅ 正确的理解                           │
│  MPPT直流输出 → 直流母线 → 电池          │
│                      ↓                   │
│                   PCS (AC/DC双向)        │
│                      ↓                   │
│                   交流母线               │
└─────────────────────────────────────────┘
```

**核心要点**:
- MPPT直接输出直流电到直流母线，**不经过PCS**
- PCS只负责AC/DC双向变换，连接交流侧和直流侧
- 光伏和电池在直流侧是**并联关系**，功率直接交换
- 直流侧功率平衡方程: `P_bat = P_clu/η - P_pv_actual`

**参考文件**:
- 拓扑图: `02-Domain-Model.md` 第2.1节（已纠正版本v1.2+）
- 状态空间模型: `02-Domain-Model.md` 第2.5节
- 学术分析: `references/MPC_ESS_Academic_Analysis.md`

**如果我忘记了，用户会生气。** 请务必先读02-Domain-Model.md的拓扑部分。

## 6. 待确认事项管理机制

### 6.1 为什么需要
项目推进过程中会产生大量待确认的技术参数、待补充协议、待决策事项。分散记录容易遗漏，需要集中管理。

### 6.2 记录位置
`projects/cot/docs/ai-dev/PENDING-ITEMS.md`

### 6.3 记录规则
1. **分类记录** - 按"协议文档/技术参数/文档待完善"分类
2. **标注优先级** - 高/中/低，高优先级优先确认
3. **打勾更新** - 用户确认后立即更新，标注确认日期
4. **定期归档** - 已确认事项可移至"历史确认"区域

### 6.4 何时读取
- **每次启动** - 了解当前待确认事项状态
- **用户询问** - 当用户问"cot还有什么问题"时，直接引用本文件
- **会话结束前** - 确认是否有新问题需要加入

### 6.5 如何响应用户询问
当用户问"cot还有什么待确认的"或类似问题时：
1. 读取 `PENDING-ITEMS.md`
2. 按优先级列出未确认事项
3. 询问用户想确认哪些
